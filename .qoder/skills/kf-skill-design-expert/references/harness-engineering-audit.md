# Harness Engineering 五根铁律 — 完整评审体系

> **使用方法**：将此文件加载到 AI 上下文中，配合 `harness-audit.cjs` 脚本，即可对任意项目进行全路径 Harness Engineering 评审。

---

## 一、五根铁律总览

| # | 铁律 | 核心问题 | 一句话 |
|---|------|---------|--------|
| 1 | **指令** | AI 知道做什么、怎么做吗？ | 地图而非百科全书 |
| 2 | **约束** | 规则能被机械化验证吗？ | 不让 AI 自己判断自己 |
| 3 | **反馈** | 产出被自动验证了吗？ | Plan→Build→Verify→Fix 闭环 |
| 4 | **记忆** | 经验被跨会话保留了吗？ | 不让 Agent 犯同样的错误 |
| 5 | **编排** | 上下文窗口被合理分工了吗？ | 子 Agent 防火墙 + 模型路由 |

---

## 二、铁律 1 — 指令（Instructions）

### 核心理念
Skill 的指令是"地图"不是"百科全书"。每条指令都必须是**可执行的、可验证的**，不说废话。Agent 看到一条指令就知道该做什么、怎么做、怎么算做完。

### 评分标准（1-5 分）

| 分 | 标准 | 特征 |
|----|------|------|
| 1 | **不可执行** | 只有笼统描述（"帮助用户写代码"），无步骤、无门控、无产出定义 |
| 2 | **模糊** | 有大致方向但缺少关键步骤，Agent 需要大量自行判断 |
| 3 | **基本可执行** | 有步骤但缺少门控条件，产出定义不够明确 |
| 4 | **可执行** | 步骤清晰、有门控、产出明确，但缺少边界条件处理 |
| 5 | **地图级** | 步骤→门控→产出→边界→异常处理全链路清晰，MUST/MUST NOT 明确 |

### 评审检查清单

- [ ] Steps 是否用数字编号，逻辑顺序正确？
- [ ] 每一步是否有明确的输入/输出定义？
- [ ] 是否有 Gate 条件阻止跳步（→ 铁律 2）？
- [ ] MUST/MUST NOT 是否具体可执行？
- [ ] 是否避免了"一个 Skill 做太多事"？
- [ ] 是否有 Gotchas 章节记录特有陷阱？
- [ ] 每条指令能否回答"Agent 没有这条会做错吗"？（删除通用常识）
- [ ] 联动调用（integrated-skills）是否明确声明了触发时机？

### 常见缺陷

| 缺陷 | 示例 | 正确做法 |
|------|------|---------|
| 笼统描述 | "帮助用户做代码审查" | "Step 1: 读取目标文件 → Step 2: 逐项检查 review-checklist.md → Step 3: 输出分级报告" |
| 缺少门控 | "生成文档" | "Gate 1: 所有 Phase 1 问题回答完毕前 MUST NOT 生成文档" |
| 冗余常识 | "使用正确的语法" | 删除——Agent 已经知道 |
| 联动未声明 | 文档产出后不触发下游技能 | frontmatter 声明 `integrated-skills`，在合适步骤写明 `Skill({...})` 调用 |

## 三、铁律 2 — 约束（Constraints）

### 核心理念
约束必须**机械化执行**，不能依赖 AI 的自觉。用 lint/CI/脚本/hook 等自动化手段强制验证，让机器检查机器。

### 评分标准（1-5 分）

| 分 | 标准 | 特征 |
|----|------|------|
| 1 | **无约束** | 没有任何机械化验证手段 |
| 2 | **仅有文字约束** | 写了 MUST/MUST NOT 但没有对应的检查机制 |
| 3 | **部分机械化** | 有 harness-gate-check 调用但未覆盖全部关键门控 |
| 4 | **基本机械化** | 关键门控有机械化验证，但缺少阻断逻辑 |
| 5 | **全机械化** | 所有门控有 hook/脚本验证，验证失败阻断后续流程 |

### 评审检查清单

- [ ] 每个 Gate 是否有对应的机械化验证命令（`harness-gate-check.cjs`）？
- [ ] 验证失败是否**阻断**后续流程（非仅警告）？
- [ ] 门控验证是否覆盖关键产出（required-files / required-sections / forbidden-patterns）？
- [ ] 是否有 `--forbidden-patterns TODO 待定` 等禁止模式检查？
- [ ] Hook 配置（settings.json）是否包含该验证步骤？
- [ ] 同一 Gate 失败 2 次是否有升级机制（→ 铁律 4）？

### 机械化验证手段

| 手段 | 工具 | 适用场景 |
|------|------|---------|
| Gate 验证 | `harness-gate-check.cjs` | 门控条件检查 |
| Lint/Typecheck | ESLint, tsc, go vet | 代码质量 |
| 测试套件 | Jest, pytest, go test | 功能正确性 |
| Hook 自动触发 | PreToolUse/PostToolUse | 操作前后自动检查 |
| 格式检查 | Prettier, black, gofmt | 代码格式 |

## 四、铁律 3 — 反馈（Feedback）

### 核心理念
产出不能靠"我觉得好了"，必须走 **Plan → Build → Verify → Fix** 四步强制循环。每步产出后立即验证，不积累问题到下一阶段。

### 评分标准（1-5 分）

| 分 | 标准 | 特征 |
|----|------|------|
| 1 | **无反馈** | 产出后无任何验证步骤 |
| 2 | **声明式检查** | "确认无误"就算通过，无实际验证动作 |
| 3 | **有验证但无闭环** | 每个 Step 有验证但失败后无修复→再验证循环定义 |
| 4 | **基本闭环** | 有 Plan→Build→Verify→Fix 循环，但部分步骤缺少验证 |
| 5 | **完整闭环** | 每个 Step 有验证动作 + 失败回退路径 + 重新验证机制 |

### 评审检查清单

- [ ] Skill 中是否定义了每个 Step 的验证动作？
- [ ] 验证动作是否具体可执行（不是"确认功能正常"）？
- [ ] 是否有 `Plan → Build → Verify → Fix` 循环定义？
- [ ] 是否拒绝了"主观觉得好了"式的验收？
- [ ] 是否有路由决策/状态变更的统计记录（→ 铁律 4）？
- [ ] 是否有 per-step 的验证命令表格（类似 kf-spec 的 Harness 反馈闭环章节）？

### 反馈闭环模板

每个流程型 Skill 应包含：

```markdown
## Harness 反馈闭环

| Step | 验证动作 | 失败处理 |
|------|---------|---------|
| Step N | `harness-gate-check.cjs --required-files "xxx.md"` | 回退补充 |
| Step M | typecheck/lint | 阻断后续 |
```

## 五、铁律 4 — 记忆（Memory）

### 核心理念
不让 Agent 犯同样的错误。失败教训、成功模式、重复决策都应跨会话持久化。每次启动自动加载历史基线。

### 评分标准（1-5 分）

| 分 | 标准 | 特征 |
|----|------|------|
| 1 | **无记忆** | 每次会话从零开始，无任何持久化 |
| 2 | **仅有 MEMORY.md** | 有索引但内容稀疏，无结构化记录 |
| 3 | **有日志但无回读** | 记录了但下次不加载，等同没记 |
| 4 | **基本闭环** | 有记录 + 下次加载 N 条历史作为基线 |
| 5 | **完整记忆系统** | 结构化记录 + 自动加载基线 + 统计分析 + 过期清理 |

### 评审检查清单

- [ ] 是否有 Harness 记忆持久化章节（如 kf-alignment 的 alignment-log.md）？
- [ ] 是否在 Skill 启动时自动读取历史基线（如"最近 3 条记录"）？
- [ ] 记忆格式是否结构化（date、skill、stage、decision、issue）？
- [ ] 是否有统计汇总（如 kf-model-router 的 routing-stats.md）？
- [ ] 是否有 MEMORY.md 索引文件维护指南？
- [ ] 旧记录是否有过期/清理策略？

### 记忆类型与存储位置

| 记忆类型 | 存储位置 | 示例 |
|---------|---------|------|
| 对齐记录 | `memory/alignment-log.md` | kf-alignment 每次对齐的结构化记录 |
| 路由统计 | `memory/model-routing-stats.md` | kf-model-router 模型切换统计 |
| 评分结果 | `memory/mvp-results.md` | kf-mvp 历史评分卡 |
| 项目上下文 | `memory/` 目录下的项目级记忆 | 用户角色、偏好、约束 |

## 六、铁律 5 — 编排（Orchestration）

### 核心理念
上下文窗口是稀缺资源。通过子 Agent 分工建立**上下文防火墙**，通过模型路由让思考型任务用 pro、执行型任务用 flash。协调者做决策，执行者做落地。

### 评分标准（1-5 分）

| 分 | 标准 | 特征 |
|----|------|------|
| 1 | **无编排** | 所有工作在一个 Agent 内完成，上下文堆积 |
| 2 | **有分工但无模型路由** | 拆了 Agent 但都用同一模型 |
| 3 | **基本路由** | frontmatter 声明了 recommended_model，但 agent 级未区分 |
| 4 | **Agent 级路由** | 协调者 pro + worker flash，通过 Agent 工具的 model 参数控制 |
| 5 | **完整编排** | 上下文防火墙 + agent 级模型路由 + 门控触发的自动切换 + ROI 统计 |

### 评审检查清单

- [ ] 是否声明了 `recommended_model`（计划/审查用 pro，执行/文档用 flash）？
- [ ] 子 Agent spawn 是否指定了 model 参数？
- [ ] 协调者（决策/评判）是否用 pro 级别模型？
- [ ] 执行者（编码/测试/设计）是否用 flash 级别模型？
- [ ] 是否有模型切换的统计记录（→ 铁律 4）？
- [ ] integrated-skills 是否声明了完整的联动链条？
- [ ] 门控触发是否会自动调用模型路由（→ 铁律 2）？

### 模型路由原则

| 阶段 | 模型 | 原因 |
|------|------|------|
| 计划/设计 | pro | 深度推理、架构决策 |
| 执行/编码 | flash | 效率优先 |
| 代码审查 | flash | 模式匹配为主 |
| Bug 排查 | pro | 深度上下文理解 |
| 文档生成 | flash | 结构化输出 |

## 七、评审方法论

### 评审流程

```
1. 全路径扫描
   ├─ 扫描所有 kf-*/SKILL.md 文件
   ├─ 提取 frontmatter、Steps、Gates、Harness 章节
   └─ 检查联动声明完整性

2. 逐铁律评分
   ├─ 铁律 1（指令）：检查 Step 清晰度、门控、MUST/MUST NOT
   ├─ 铁律 2（约束）：检查机械化验证命令、Hook 配置
   ├─ 铁律 3（反馈）：检查 Plan→Build→Verify→Fix 闭环
   ├─ 铁律 4（记忆）：检查 memory/ 目录、自动加载基线
   └─ 铁律 5（编排）：检查 recommended_model、agent model 参数

3. 缺陷分级
   ├─ P0（阻塞）：核心流程缺失，AI 无法正确执行
   ├─ P1（重要）：影响质量/效率，应尽快修复
   └─ P2（优化）：锦上添花，有时间再做

4. 输出报告
   ├─ 技能 × 铁律评分矩阵
   ├─ 缺陷清单（分级 + 责任人 + 修复建议）
   ├─ 系统性缺陷分析
   └─ 改进优先级建议
```

### 自动化扫描命令

```bash
# 全量审计
node {IDE_ROOT}/helpers/harness-audit.cjs --all

# 单技能审计
node {IDE_ROOT}/helpers/harness-audit.cjs --skill kf-mvp

# 单铁律审计（如只查"约束"）
node {IDE_ROOT}/helpers/harness-audit.cjs --rule constraints

# 输出 JSON 格式
node {IDE_ROOT}/helpers/harness-audit.cjs --all --format json
```

## 九、持续改进

### 评审节奏

| 频率 | 动作 | 负责人 |
|------|------|--------|
| 每次技能变更 | 单技能审计（`--skill xxx`） | 开发者 |
| 每周 | 全量审计（`--all`） | AI (cron) |
| 每月 | 报告趋势分析 + 铁律迭代 | 团队 |

### 历史记录

审计结果自动持久化到 `memory/harness-audit-history.md`，格式：

```markdown
### {date} — 全量审计

| 技能 | 指令 | 约束 | 反馈 | 记忆 | 编排 | 均分 |
|------|------|------|------|------|------|------|
| ... | ... | ... | ... | ... | ... | ... |

**本次发现新缺陷**：N 个（P0: X, P1: Y, P2: Z）
**较上次变化**：均分 +0.X
```
