# doc-ultra 文档超融合处理

## 触发方式

用户消息包含以下任一关键词时自动激活：
- `doc-ultra`
- `文档处理` `文档优化` `文档检查` `文档扩写`
- `优化文档` `检查文档` `润色文档`
- 明确要求对标书/专利/软著/团标/申报书/知识库文档做处理

## 执行方式

首选：使用已安装的 `doc-ultra` CLI 工具（一条命令完成全流程）。
次选：手动按流水线步骤依次执行（CLI 不可用时）。

---

## 方式一：CLI 一键执行（推荐）

```bash
# 完整流水线（指定文档类型预设）
python -m doc_ultra.cli <输入文件> -p <预设> -o <输出文件>

# 自动检测文档类型
python -m doc_ultra.cli <输入文件> --auto-preset -o <输出文件>

# 仅检查
python -m doc_ultra.cli <输入文件> --check-only

# 带扩写
python -m doc_ultra.cli <输入文件> -p bid --target-words 5000 -o 输出.md

# 查看可用预设
python -m doc_ultra.cli --list-presets
```

**预设对照**（详见 [presets.md](presets.md)）：

| 预设名 | 适用场景 |
|--------|---------|
| `bid` | 招投标文档 |
| `patent` | 专利文档 |
| `software_copyright` | 软件著作权 |
| `standard` | 团体/行业标准 |
| `project_application` | 项目申报书 |
| `knowledge_base` | 基于知识库的文档 |
| `general` | 通用文档 |
| `economical` | 低成本模式 |

## 方式二：手动流水线（CLI 不可用时）

按以下 6 个 Stage 依次执行。每步产出写入 `.doc-ultra/` 目录。

### Stage0 — 需求解析

读取用户提供的原始文档 + 附件，产出**需求规格书**。

执行方式：在对话中让 Agent 分析文档，提取以下结构化信息并写入 `.doc-ultra/stage0-requirement-spec.md`：

```markdown
# 需求规格书
## 1. 文档元信息（用途/受众/篇幅）
## 2. 内容范围（必须覆盖的章节/主题 + 边界）
## 3. 格式约束（层级/风格/术语）
## 4. 附件信息提取
## 5. 质量检查标准（显式 + 隐式）
```

### Stage1 — 多视角并行优化

用 **3 个不同价值主张** 的视角分别优化原文，产出 3 份独立草稿。

> 使用 Qoder 的多 Agent 并行能力（Agent 工具），同时启动 3 个子 Agent，各自使用不同的 system prompt：

| 视角 | 价值主张 | 系统提示词要点 |
|------|---------|---------------|
| 激进派 | 追求极致质量 | 大幅调整结构、替换平庸表述、积极补充缺失内容 |
| 保守派 | 最小改动 | 保持原文结构、仅修正明确问题、谨慎补充 |
| 均衡派 | 务实改进 | 选择性重构、适中润色、合理补充 |

每个 Agent 收到：需求规格书 + 原始文档，产出完整优化稿。产出文件命名：`.doc-ultra/stage1-draft-{激进/保守/均衡}.md`

### Stage2 — 融合合成

读取 3 份优化稿，以保守派为骨架，吸收激进派结构优化和均衡派措辞优化，产出融合稿。

执行方式：单 Agent 任务，system prompt 参考 [reference.md](reference.md) §Stage2。

产出：`.doc-ultra/stage2-draft-fused.md`

### Stage3 — 拷问检查循环（↺ 核心质量控制）

对照需求规格书，按四维标准逐项检查融合稿：

| 维度 | 通过条件 |
|------|---------|
| 内容覆盖度 | 100% 覆盖需求规格书要求的主题 |
| 格式合规度 | 偏差 ≤ 2 处 |
| 逻辑自洽度 | 0 处矛盾 |
| 需求忠实度 | 0 处实质性偏离 |

执行方式：
1. Agent 检查融合稿 → 产出检查报告
2. 不通过 → 根据报告修复 → 重新检查
3. 最多 3 轮，超则标记未决问题交由用户决策

产出：`.doc-ultra/stage3-draft-checked.md` + 各轮检查报告

### Stage4 — 扩写（可选）

仅当用户明确要求时执行。约束：**不变实质内容**，仅补充例证/细化描述/增加过渡。

产出：`.doc-ultra/stage4-draft-expanded.md`

### Stage5 — 终审抛光

执行四步：
1. **AI 痕迹消除**：移除「在当今时代」「综上所述」「值得注意的是」等高频 AI 句式；打破段落对称结构；去模板化开头
2. **MD 转 doc 兼容**：简化复杂表格、统一标题层级
3. **格式统一**：列表样式、引用格式、空行间距
4. **最终扫读**：无残留占位符/TODO/语法错误

产出：`output.md`（最终交付稿）

## 模型分配策略

各阶段按"质量敏感用高端模型，数量敏感用性价比模型"原则分配：

| 阶段 | 策略 | 推荐模型 |
|------|------|---------|
| Stage0 解析 | 需精确理解 → 高端 | GPT-4o / Claude |
| Stage1 优化 | 并行 3 份 → 性价比 | Claude(激进) + GPT-4o(保守) + DeepSeek(均衡) |
| Stage2 融合 | 需推理+长文 → 高端 | Claude |
| Stage3 检查 | 质量底线不可降 → 高端 | GPT-4o |
| Stage4 扩写 | 量大 → 性价比 | DeepSeek / Kimi |
| Stage5 抛光 | 快+便宜 → 快速模型 | GPT-4o-mini |

## 文件约定

中间产物全部存放在 `.doc-ultra/` 目录，最终输出由用户指定的路径或默认 `./output.md`。

完成流水线后**必须向用户报告**：
- 每个 Stage 的耗时
- 检查循环轮次和最终判定
- 输出文件路径
- 若使用了预设，注明预设名称和分配策略
