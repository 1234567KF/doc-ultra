# 技术规范文档编制常见错误与防范清单

> 适用场景：AI辅助编制行业标准、技术规范、团体标准、投标技术文件等正式文档。
> 本文档抽象自航空追溯项目三规范编制的多轮修订实践，目标是前置规避同类错误。

---

## 一、编号体系

### 错误1：正文改了编号，附录/交叉引用未同步

**表现**：章节从"第十三条"改为"3.7"后，附录中"按第十三条算法生成"未改，或错误地改成了"按1.4"。

**根因**：编号修改时只改了标题，未全文搜索交叉引用文本。

**防范**：
- 每次修改章节编号后，**全文grep**以下模式：
  - `第[一二三四五六七八九十]+[章条款]`（中文数字引用）
  - `按.*算法`、`参见.*[0-9]`、`如.*[0-9]\.[0-9]`（隐性引用）
  - 数字编号的松散引用（如"见1.4"）
- 修改完正文编号后，**立即**检查附录、示例、注释中的所有引用。

### 错误2：四级标题编号与父级脱节（15.1  ≠ 4.2.1）

**表现**：`## 4` → `### 4.2` → `#### 15.1`，15.1本应在4.2之下却编号为15.1。

**根因**：AI按"全文顺序自增编号"，而非"按层级嵌套编号"。

**防范**：
- **四级标题始终按父级嵌套**：`章.条.款` = `# 4` → `## 4.2` → `### 4.2.1`
- 生成文档后执行验证：`grep '#### [0-9]+\.[0-9]+'` 检查所有四级标题编号是否与其上一个`###`标题的编号前缀一致。

### 错误3：混淆"章/条"中文编号与GB/T 1.1数字编号

**表现**：标准文档使用"第一章/第一条"而非"1/1.1"，不符合国家标准格式。

**根因**：未识别文档类型（标准 vs 普通报告），采用了错误的编号规范。

**防范**：
- **先确定文档类型再定编号规范**：
  - 行业标准/国标 → GB/T 1.1（阿拉伯数字层级编号）
  - 普通报告/方案 → 可保留中文编号
- 对于标准类文档，**生成后立即执行编号转换脚本**，将"第X章/第X条"转为"X/X.Y"。

## 二、编码/数据结构一致性

### 错误4：字段位数变更后，所有下游数据失效

**表现**：批次码从6位改为5位、序列码从6位改为7位后，校验码计算示例、附录UID示例、GJB 726A映射描述全部过时。

**根因**：字段位数是全局约束，但修改时只改定义表，未追溯所有依赖。

**防范**：
- **字段位数修改 = 三级检查**：
  1. **定义层**：结构表、字段说明、编码规则
  2. **示例层**：校验码计算过程（逐位加权表、和值、模值）、附录完整UID示例
  3. **衔接层**：与其他标准的映射关系表、外部系统接口描述
- 修改字段位数后，**必须重新手算校验码**（不能复用旧值），并更新所有示例。

### 错误5：原文档自带位序bug（如型号码"5-8位"实际只有6位）

**表现**：型号码6位字段写"1-4位 + 5-8位"（8超出6），配套描述"0000"也错。

**根因**：AI生成时未做"位序总和 = 字段位数"的自洽校验。

**防范**：
- 所有编码规则表的位序行，**验证：末位 ≤ 字段总位数**。
- 配套描述中的占位字符数（如0000=4位）应与实际分配位数一致。

## 三、去品牌化/中立性

### 错误6：企业专属术语、指标混入标准文档

**表现**：最小面积"≤0.5mm×0.8mm"（微点码独有非正方形指标）、"外部发码模式"（基于微点视角）、"发码中心"（企业术语）。

**根因**：AI直接从知识库提取企业技术参数，未做中立化转换。

**防范**：
- **生成前先执行术语扫描**：列出文档中所有数值型指标和有品牌倾向的术语。
- **规则**：
  - 非正方形物理参数 → 取长宽较大值改为正方形（如0.5×0.8 → 0.8×0.8）
  - "发码中心" → "编码管理中心"
  - "外部发码模式" → "分布式发码模式"
  - 禁止将企业自有技术规范列为编制依据
- **编制依据只列国标、行标、国军标、ISO/IEC标准**。

### 错误7：标准引用幻觉

**表现**：引用了不存在或编号/名称错误的标准（如GJB 5793/5794/5795、GB/T 22005-2009等）。

**根因**：AI根据训练数据生成标准引用，未做真实性验证。

**防范**：
- **所有引用的标准必须逐条核查**：
  1. 通过搜索引擎确认标准编号+名称是否真实存在
  2. 确认发布单位（GB→国标委、GJB→军委装发部、HB→航空工业、ISO/IEC→国际）
  3. 删除无法确认的标准引用
- 宁可少列，不可列错。

## 四、映射关系描述

### 错误8："格式统一为XXXX"误导为格式等同

**表现**：GJB 726A批次号是YYYYMMDD+NN（10位），本规则批次码是YYMMB（5位），却描述为"格式统一为YYYYMMDD+NN"。

**根因**：混淆了"映射关系"和"格式统一"两个概念。

**防范**：
- 与外部标准的衔接表，使用**映射语言**而非统一语言：
  - ✅ "提取年份末两位+月份+批次序号，映射为YYMMB格式"
  - ❌ "格式统一为YYYYMMDD+NN"
- **检查每条映射描述是否能从来源字段的格式推导出目标字段的格式**——如果不能，说明描述有误。

## 五、DOCX生成质量

### 错误9：模板继承污染——文档保护导致不可编辑

**表现**：生成的DOCX无法编辑，因为模板文档带有`documentProtection`。

**根因**：Pandoc使用`--reference-doc`时会继承模板的全部XML设置，包括保护标记。

**防范**：
- 转换脚本后处理中**主动移除documentProtection**：
  ```python
  settings = doc.settings.element
  ns = '{http://schemas.openxmlformats.org/wordprocessingml/2006/main}'
  for prot in settings.findall(f'{ns}documentProtection'):
      settings.remove(prot)
  ```

### 错误10：列表缩进过大、分割线残留、代码块灰色背景

这些已在kf-markdown-to-docx脚本层面修复，此处不赘述。

## 六、跨文档一致性

### 错误11：一个文档的底层结构变更未检查其他文档

**表现**：编码规则改了UID结构（批次码和序列码位数），未检查数据采集规范、安全管理规范中是否有对该结构的引用。

**防范**：
- 任何**底层架构变更**（数据结构、编码规则、关键术语），跨文档grep受影响文档：
  ```
  grep "批次码|序列码|32位|28位|UID结构" 其他文档/*.md
  ```

## 七、执行检查清单（每次生成/修改后必做）

| 检查项 | 方法 |
|--------|------|
| 所有章节编号符合GB/T 1.1（无"第X章"残余） | `grep "第[一二三四五六七八九十]+[章条款]" *.md` |
| 四级标题编号与父级一致 | `grep "#### [0-9]" *.md` 人工比对 |
| 附录引用全部有效 | `grep "按.*[0-9]\.[0-9]" *.md` 逐一核实 |
| 编码结构表位序自洽 | 末位 ≤ 字段位数 |
| 校验码示例完整且与新结构一致 | 手算一遍 |
| 外部标准映射描述是"映射"非"统一" | `grep "格式统一" *.md` |
| 无品牌化术语 | `grep "微点\|发码中心\|外部发码" *.md` |
| 标准引用全部可查证 | 逐条搜索引擎核实 |
| 跨文档无断裂引用 | 结构变更后grep其他文档 |
| DOCX可编辑 | 检查documentProtection已移除 |
